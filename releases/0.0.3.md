# TrenchClaw v0.0.3

`0.0.3` is a release about making the runtime feel more legible while it grows more capable.

The visible theme is better operator awareness: queued trading work is easier to reason about, runtime prompts have a clearer sense of live state, wakeup behavior keeps getting more explicit, and workspace tooling is being pushed toward clearer real-shell usage instead of vague pseudo-agent behavior.

The less visible theme is discipline.

This patch continues the shift toward:

- explicit runtime modules
- typed scheduling and action boundaries
- better prompt/runtime agreement
- Bun-first release engineering
- release notes that actually show what shipped

That matters because TrenchClaw is at the stage where product trust is shaped less by giant headline features and more by whether the runtime, prompts, tools, tests, and release surface all tell the same story.

`0.0.3` moves the repo further in that direction.

## Executive Summary

If you only read one section, read this one.

`0.0.3` improves TrenchClaw in five practical ways:

1. It makes scheduled trading work easier to inspect and reason about through upcoming-job views and live runtime context improvements.
2. It continues tightening trading routine and scheduling infrastructure so queued execution is more explicit and testable.
3. It improves workspace CLI tooling and prompt guidance so the model is steered toward real shell behavior and safer directory traversal patterns.
4. It keeps cleaning up prompt, wakeup, and knowledge-index plumbing so the system prompt and the actual runtime surface stay aligned.
5. It fixes the release surface itself by making release notes long-form again and by replacing the stale README version badge with a badge that follows the latest GitHub release.

None of that is flashy for its own sake.

All of it is high leverage.

## Release Highlights

- Added better runtime visibility for upcoming scheduled trading work, including store-query support and live prompt context that can surface queued jobs.
- Expanded and tightened scheduling utilities so interval, delay, and time-based trading flows are easier to validate and reason about.
- Improved workspace CLI/tooling guidance and traversal behavior so the runtime behaves more like a serious local operator environment and less like a fuzzy assistant shell.
- Continued refining wakeup, prompt, routing, and knowledge-surface behavior so the model sees a cleaner description of the live system.
- Fixed the public release surface itself: `0.0.3` is set up for a proper long-form release body, and the README release badge now tracks the latest GitHub release automatically instead of hardcoding an old version.

## Why `0.0.3` Matters

`v0.0.2` improved reliability around release packaging and runtime/logging details.

`0.0.3` is where the runtime starts getting better at answering a deeper operational question:

What is actually going on right now, and what is going to happen next?

That question matters more as the system gets more scheduling behavior, more action surface, and more agent-facing tooling.

Without that visibility, you end up with a repo that can technically do more things, but is harder to trust because too much of the important state is implicit.

This release pushes in the opposite direction.

It makes more of the runtime legible:

- upcoming trading jobs can be surfaced intentionally
- schedule parsing and validation are more explicit
- prompt context can describe queued trading work instead of acting blind
- workspace CLI behavior is guided more directly
- release artifacts and release messaging are less likely to drift away from reality

That is exactly the kind of patch release that makes future feature work safer.

## The Main Themes Of This Release

### 1. Better visibility into scheduled trading work

One of the most important improvements in this patch is not a brand-new trading primitive. It is better runtime awareness of already-scheduled trading behavior.

That includes:

- upcoming trading job extraction from runtime state
- query support so the runtime store can return queued trading jobs directly
- live prompt context that can surface scheduled trades when the model needs to answer operational questions

This matters because once trading work can be queued, staggered, or delayed, operators need a clean answer to questions like:

- what is still pending
- what is next
- what kind of routine is scheduled
- what wallet, pair, and amount a queued action represents

Surfacing that information is the difference between "automation exists somewhere in the system" and "the operator can actually manage it."

### 2. Scheduling is becoming a real subsystem instead of scattered timing logic

Another major theme in `0.0.3` is the continued formation of dedicated scheduling infrastructure.

This release strengthens the path toward:

- normalized duration parsing
- explicit granularity rules
- derived interval calculation
- better upcoming-job summaries
- more test coverage around schedule behavior

That kind of work is easy to underestimate because it can look like plumbing.

But in a runtime like this, scheduling plumbing is product surface.

If timing behavior is ambiguous, every delayed trade, DCA path, wakeup cycle, or queued action becomes harder to trust. Tightening that layer is one of the cleanest ways to improve reliability without pretending the system is more mature than it is.

### 3. Workspace CLI behavior is being treated more seriously

The workspace-bash and workspace-directory tooling changes matter because they affect how the runtime approaches the user’s actual machine.

This release continues the shift away from loose "agent shell" vibes and toward a clearer model:

- the runtime should prefer real shell and CLI usage
- directory traversal should be stable and predictable
- routing hints should steer the model toward the right workspace tools
- filesystem access should be framed as explicit operator access, not hand-wavy magic

That is the correct direction for TrenchClaw.

This project should feel like a local operator runtime with typed tools and deliberate boundaries, not like an improvisational bot that sometimes wanders through the filesystem.

### 4. Prompt and knowledge cleanup is still compounding

Another important part of this patch is continued cleanup across:

- operator prompt text
- lane policy/routing hints
- live runtime context
- wakeup wiring
- knowledge index and knowledge-surface behavior

That cleanup matters because prompt quality in this project is not just "AI prompt engineering."

It is runtime design.

When the prompt says the wrong thing about the system, operators pay for it in confusion, bad tool choices, and unnecessary retries. When the prompt accurately reflects the runtime, the model behaves more like a usable control surface.

`0.0.3` keeps investing in that alignment.

### 5. The release surface now matches your expectations again

This patch also fixes something much more visible: release presentation.

Recent automation was publishing GitHub-generated notes that were too short and often collapsed into little more than a changelog link.

That is not the kind of release surface this project wants.

For `0.0.3`, the workflow is set up so release notes can be long and useful again:

- manual `draft` releases can ship a narrative markdown file like this one
- the workflow appends a grouped auto-generated commit appendix after the narrative
- manual `auto` releases still publish the full commit appendix instead of GitHub’s thin autogenerated summary
- the README badge now follows the latest GitHub release automatically, so it will not get stuck on `0.0.0` again

This is a release-engineering fix, but it is also a product fix.

The public release page is part of the product surface.

## What Operators Actually Get In `0.0.3`

After this release, the TrenchClaw story is better described as:

- a local operator runtime with more visibility into queued trading behavior
- a runtime that is becoming more explicit about time, schedule, and pending work
- a tool surface that keeps steering toward real CLI usage instead of ambiguous assistant behavior
- a prompt/runtime pair that is closer to the actual live system state
- a release surface that better reflects what changed and why

That is a more credible story than simply saying "there are more tools now."

It is also more useful.

## What This Release Does Not Pretend

`0.0.3` does not suddenly make TrenchClaw:

- a fully autonomous strategy engine
- a complete portfolio management suite
- a polished mainstream end-user product
- a risk-free trading system

It does make the system easier to inspect, reason about, and operate.

That distinction matters.

Good release notes should separate:

- what is newly shipped
- what is newly clearer
- what is newly more trustworthy
- what is still intentionally not claimed

This release is trying to do that honestly.
