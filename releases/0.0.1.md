# TrenchClaw v0.0.1

`0.0.1` is the first real cleanup-and-hardening patch after `v0.0.0`. This release is not just a small docs touch-up or a version bump for the sake of it. It meaningfully improves the operator surface, expands the shipped Solana action set, cleans up runtime/storage internals, and makes the public-facing product story much more coherent.

The biggest theme in this release is consolidation. TrenchClaw now does a better job of matching what the repo says, what the runtime exposes, what the GUI presents, and what the codebase is actually organized around. The result is a release that feels less like a fast-moving prototype stitched together from multiple iterations and more like a deliberate operator runtime with cleaner boundaries.

This patch also continues the larger direction of the project: Bun-first tooling, TypeScript-first orchestration, `@solana/kit`-native execution, and a constrained runtime model that favors explicit modules, typed actions, and auditable state over vague autonomy.

## Release Highlights

- Added Jupiter Trigger order support to the runtime, including order creation, reads, and cancellation flows.
- Added wakeup settings plumbing across the GUI and runtime API so more runtime behavior is visible and configurable through the operator surface.
- Added a rate-limited RPC client and improved managed wallet read paths.
- Refactored logging and storage internals to make runtime persistence cleaner and less fragmented.
- Tightened prompt/runtime language and renamed the old "Runtime Contract" framing to "Live Runtime Rules".
- Cleaned up the README surface in English and Japanese, including a better product screenshot and a more focused lower-half structure.

## Why `0.0.1` Matters

`v0.0.0` established the first public release path. `0.0.1` is where that public release starts to feel more honest, maintainable, and operationally usable.

This matters for a few reasons:

- The runtime surface is broader in a useful way, especially around Trigger orders and wakeup/runtime controls.
- The internals are cleaner in areas that directly affect maintainability: storage layout, log I/O, runtime state handling, wallet context, and capability wiring.
- The docs and release surface now present the product more clearly, which reduces mismatch between what users expect and what the app actually ships today.

This is still experimental software. That warning has not changed. What has changed is that more of the project now lines up cleanly across code, docs, UI, and release messaging.

## Operator-Facing Improvements

### Jupiter Trigger order support

One of the most meaningful shipped expansions in this patch is the addition of Jupiter Trigger order support.

This includes:

- Trigger order creation flows
- Managed Trigger order creation
- Trigger order listing / reads
- Trigger order cancellation
- Shared Trigger action logic and adapter support
- Runtime capability registration and test coverage around the new surface

Why this matters:

- TrenchClaw moves closer to being a practical operator runtime rather than only a direct-execution swap shell.
- Operators can work with a broader class of swap workflows using the same typed action and policy-oriented runtime structure.
- The Trigger support is integrated as an explicit module set, not bolted on as an opaque side-path.

This work is especially important because it extends the runtime in a direction that fits the project’s design language: explicit actions, constrained execution, typed boundaries, and testable adapters.

### Wakeup settings UI + runtime transport

Another visible improvement in this release is the addition of wakeup settings support across the GUI/runtime boundary.

This release adds:

- A new `WakeupPanel` in the GUI workspace
- Runtime API support for wakeup-related settings
- New runtime loading and settings I/O paths for wakeup configuration
- Related instance-path and settings schema updates

Why this matters:

- More runtime behavior is exposed through the actual operator surface instead of remaining buried in implementation details.
- The GUI continues to move toward being the real control plane for local operators.
- This reduces the gap between "runtime exists" and "runtime is actually controllable in a clean way."

### Better product presentation

The README surface now does a better job of showing the app as a usable product.

That includes:

- A refreshed main UI screenshot

## Runtime And Infrastructure Cleanup

This release includes a substantial amount of under-the-hood cleanup that should make the runtime easier to evolve.

### Rate-limited RPC client

The Solana RPC path now includes a dedicated rate-limited client layer.

Why this matters:

- It gives TrenchClaw a more explicit place to manage RPC pressure and request behavior.
- It reduces the chance that higher-level action code becomes a dumping ground for provider-specific throttling concerns.
- It improves the shape of the runtime for real-world operator workloads, especially when multiple read paths and automation flows are active.

### Managed wallet reads got more serious

Managed wallet read flows saw meaningful work, especially around `getManagedWalletContents`.

Why this matters:

- Managed-wallet introspection is one of the most operator-visible parts of the product.
- Better read-path structure makes the runtime more useful as a decision surface, not only a transaction submitter.
- It also reduces the mismatch between what the GUI/operator wants to know and what the runtime can retrieve cleanly.

### Storage and log I/O refactor

This patch does a large amount of cleanup around runtime storage and log handling.

Notable changes include:

- consolidation around shared storage helpers
- movement away from fragmented log-store implementations
- refactoring worker/client boundaries for log I/O
- clearer runtime log store composition
- cleanup around summary/system/live log store organization
- broader SQLite schema/ORM rework

Why this matters:

- Runtime persistence is one of the highest-leverage areas in an agent runtime. If storage is messy, everything above it becomes harder to trust.
- Cleaner logging/storage boundaries improve observability and future maintenance.
- This kind of cleanup usually does not show in a screenshot, but it directly affects the project’s ability to ship more features without the runtime becoming brittle.

### Capability and runtime surface simplification

This release also cleans up parts of the capability system and runtime wiring.

Examples include:

- cleanup in capability selectors and action definitions
- removal of some older release-readiness/snapshot pathways
- tighter runtime bootstrap and load-layer organization
- cleaner prompt/capability alignment for the actual shipped product

This is important because TrenchClaw’s core value is not raw feature count. It is controlled execution through a narrow, explicit surface. Cleanup in the capability layer directly improves that value proposition.

## Prompt, Rules, And Runtime Language Cleanup

One of the less flashy but highly important changes in this patch is the cleanup around how the runtime describes itself.

The headline shift here is the rename from "Runtime Contract" to "Live Runtime Rules".

Why that matters:

- "Contract" can sound static, abstract, or implementation-only.
- "Live Runtime Rules" better communicates that these are active boundaries that shape current runtime behavior.
- It is a clearer mental model for operators, contributors, and future release messaging.

Related work in this area also tightened:

- prompt/system wording
- prompt manifest/loading behavior
- AI settings transport alignment
- tests around prompt loading and chat service behavior

This is exactly the kind of cleanup that improves consistency across product, runtime, and docs without changing the app into something else.
