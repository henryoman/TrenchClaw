# TrenchClaw v0.0.1

`0.0.1` is the first real post-`v0.0.0` cleanup-and-hardening patch, but calling it a "small patch release" would undersell it.

This release materially improves the operator surface, expands the usable Solana action set, tightens runtime and storage internals, and does a much better job of making the docs, UI, runtime model, and public messaging all say the same thing. The repo feels less like a pile of promising pieces and more like a deliberately shaped local operator runtime.

The biggest theme is consolidation.

TrenchClaw now does a better job of aligning:

- what the repo claims to be
- what the runtime actually exposes
- what the GUI can surface and control
- what the internal architecture is optimized around
- what the release story honestly promises

This patch also continues the larger direction of the project:

- Bun-first tooling
- TypeScript-first orchestration
- `@solana/kit`-native execution
- explicit runtime modules instead of vague magical behavior
- typed action boundaries instead of hidden side paths
- auditable local state over fuzzy "agent memory" claims

`0.0.1` is still experimental software.

What changed is not the risk warning. What changed is that more of the project now lines up cleanly enough to feel real.

## Executive Summary

If you only read one section, read this one.

`0.0.1` makes TrenchClaw more credible in practice by doing four things at once:

1. It adds real operator-facing capability through Jupiter Trigger order support.
2. It exposes more runtime behavior through the GUI and transport layer with wakeup settings plumbing.
3. It cleans up high-leverage internals like RPC handling, storage, log I/O, and runtime/capability wiring.
4. It makes the public-facing project story clearer through better README, site, and release language.

That combination matters more than any one individual bullet.

A lot of early-stage projects can add features.

Fewer projects can add features while also making the system simpler to reason about. That is the part that makes `0.0.1` important.

## Release Highlights

- Added Jupiter Trigger order support to the runtime, including create, managed create, list/read, and cancel flows.
- Added wakeup settings support across the GUI and runtime transport so more runtime behavior is visible and configurable from the operator surface.
- Added a dedicated rate-limited RPC client layer and improved managed wallet read paths.
- Refactored logging, storage, and SQLite-oriented runtime persistence so state handling is cleaner and less fragmented.
- Renamed the old "Runtime Contract" language to "Live Runtime Rules" and tightened prompt/runtime wording around the shipped product.
- Cleaned up the README and public docs surface, including a better screenshot presentation and a more coherent setup/story flow.

## Why `0.0.1` Matters

`v0.0.0` established the first public release path.

`0.0.1` is where that public release starts feeling more honest, more maintainable, and more operationally usable.

This is the kind of patch release that improves trust.

Not because it suddenly makes everything "safe" or "finished", but because it reduces mismatch.

Mismatch is one of the most expensive problems in operator software:

- mismatch between docs and product
- mismatch between UI and runtime
- mismatch between runtime and repo structure
- mismatch between feature claims and actual shipped behavior

`0.0.1` reduces that mismatch across several layers at once.

One of the most meaningful shipped expansions in this patch is the addition of Jupiter Trigger order support.

This includes:

- Trigger order creation flows
- Managed Trigger order creation
- Trigger order listing and reads
- Trigger order cancellation
- Shared Trigger action logic
- Trigger adapter support
- Runtime capability registration for the new surface
- Tests around Trigger-related paths

Why this matters:

- TrenchClaw moves closer to being a practical operator runtime rather than only a narrower direct-execution swap shell.
- Operators can work with a broader class of swap workflows using the same typed action and policy-oriented runtime structure.
- Trigger support is integrated as an explicit module set, not bolted on as an opaque hidden side-path.
- This is a meaningful capability increase that still fits the project’s architectural language.

Why this fits TrenchClaw specifically:

- The runtime already leans toward explicit modules over hand-wavy agent magic.
- Trigger orders benefit from typed adapters and explicit lifecycle handling.
- The feature expands what operators can do without forcing the runtime into a looser, more chaotic design.

This is exactly the kind of feature expansion that makes sense for the project.

## Runtime And Infrastructure Cleanup

This release includes a substantial amount of under-the-hood cleanup that should make the runtime easier to evolve without becoming brittle.

### Rate-limited RPC client

The Solana RPC path now includes a dedicated rate-limited client layer.

Why this matters:

- It gives TrenchClaw a more explicit place to manage RPC pressure and request behavior.
- It reduces the chance that higher-level action code becomes a dumping ground for provider-specific throttling concerns.
- It improves the shape of the runtime for real-world operator workloads, especially when multiple read paths and automation flows are active.
- It provides a cleaner architectural boundary for future RPC-related tuning.

Why this is the right direction:

- provider behavior should not leak everywhere
- request-shaping belongs in dedicated infrastructure
- operator runtimes need predictable service behavior more than they need cute abstractions

### Managed wallet reads got more serious

Managed wallet read flows saw meaningful work, especially around `getManagedWalletContents`.

Why this matters:

- Managed-wallet introspection is one of the most operator-visible parts of the product.
- Better read-path structure makes the runtime more useful as a decision surface, not only a transaction submitter.
- Cleaner wallet reads reduce mismatch between what the GUI or operator wants to know and what the runtime can retrieve cleanly.
- A stronger read surface lays groundwork for future wallet-facing improvements without forcing messy one-off patches.

This is worth emphasizing:

operator confidence depends heavily on visibility.

If wallet reads feel flaky, incomplete, or awkward, the rest of the runtime immediately feels less real. Work in this area is therefore disproportionately valuable.

### Storage and log I/O refactor

This patch does a large amount of cleanup around runtime storage and log handling.

Notable changes include:

- consolidation around shared storage helpers
- movement away from fragmented log-store implementations
- refactoring worker/client boundaries for log I/O
- clearer runtime log store composition
- cleanup around summary, system, and live log store organization
- broader SQLite schema and ORM rework

Why this matters:

- Runtime persistence is one of the highest-leverage areas in an agent runtime.
- If storage is messy, everything above it becomes harder to trust.
- Cleaner logging and storage boundaries improve observability and future maintenance.
- This kind of cleanup does not show in a screenshot, but it directly affects the project’s ability to ship more features without the runtime becoming fragile.

What got better in practice:

- fewer fuzzy boundaries
- clearer responsibilities
- better foundation for future runtime state work
- less architectural drag

### Capability and runtime surface simplification

This release also cleans up parts of the capability system and runtime wiring.

Examples include:

- cleanup in capability selectors and action definitions
- removal of some older release-readiness and snapshot pathways
- tighter runtime bootstrap and load-layer organization
- cleaner prompt/capability alignment for the actual shipped product
- improved organization around GUI/runtime transport domains

Why this matters:

TrenchClaw’s value is not raw feature count.

Its value is controlled execution through a narrow, explicit surface.

Cleanup in the capability layer directly improves that value proposition because it makes the runtime easier to understand, audit, and extend.

## Prompt, Rules, And Runtime Language Cleanup

One of the less flashy but highly important changes in this patch is the cleanup around how the runtime describes itself.

The headline shift here is the rename from "Runtime Contract" to "Live Runtime Rules".

Why that matters:

- "Contract" can sound static, abstract, legalistic, or implementation-only.
- "Live Runtime Rules" better communicates that these are active boundaries shaping current runtime behavior.
- It is a clearer mental model for operators, contributors, and future release messaging.
- It better matches how TrenchClaw actually works as a constrained runtime.

Related work in this area also tightened:

- prompt and system wording
- prompt manifest and loading behavior
- AI settings transport alignment
- tests around prompt loading and chat service behavior

This is the kind of cleanup that improves consistency across product, runtime, and docs without changing the app into something else.

That kind of consistency matters a lot in AI-adjacent software, where vague framing can quickly create false expectations.

## Wallet Surface Direction

`0.0.1` includes meaningful improvement to managed wallet reads and wallet-related runtime structure, but it should not overclaim beyond that.

The broader cleanup effort to make wallet-facing "get wallet" style actions cleaner, more coherent, and more real in day-to-day use is actively in progress.

That work is important because wallet actions are one of the places where operator expectations become extremely concrete:

- users expect the runtime to know what a wallet is
- users expect the read surface to be clean
- users expect the action names and returned data to make sense immediately
- users expect wallet operations to feel like core product behavior, not side utilities

That direction is aligned with this release, but it is intentionally not being counted as shipped inside `0.0.1` until it is actually ready.

That distinction matters.

Good release notes should separate:

- what is shipped now
- what is improved now
- what is clearly being prepared next

This release note is doing that on purpose.
