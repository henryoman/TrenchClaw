# TrenchClaw v0.1.0-beta.0

`0.1.0-beta.0` is the release where TrenchClaw starts reading like the product it already is.

The headline is not one giant new gimmick. It is coherence.

This beta brings the runtime, GUI, docs, packaging, and release surface into much tighter agreement around one clear model:

- TrenchClaw is a local runtime first
- state is instance-scoped
- tools are explicit, typed, and lane-aware
- the GUI operates the runtime instead of replacing it
- release artifacts are filtered and verified like real product builds

That matters more than any single feature checkbox, because this project is now at the point where trust comes from the shape of the system as much as the size of the surface area.

## Executive Summary

If you only read one section, read this one.

`0.1.0-beta.0` makes TrenchClaw better in five practical ways:

1. It cleans up the runtime/tool architecture so the shipped surface is easier to reason about and harder to misdescribe.
2. It tightens instance-scoped state, settings authority, and seed/runtime boundaries so the local runtime model is much clearer.
3. It expands the useful operator surface with better wallet reads, tracker/news registries, and market-analysis tools that fit the current shipped product.
4. It improves the GUI and public docs so setup, keys, settings, and architecture all tell the same story.
5. It hardens the release pipeline so packaged artifacts are filtered, verified, smoke-tested, and checked for local-path and seed-vault leakage before shipping.

This is the right beta move.

It is not pretending the product is finished. It is making the current product more legible, more credible, and safer to distribute.

## Release Highlights

- Reworked the runtime around a cleaner tool model: registry, snapshot, visibility, lane routing, and chat binding are now easier to follow end to end.
- Continued the shift from generic capability language to explicit tool language across runtime code, tests, and docs.
- Tightened instance layout and state boundaries around the seeded `00` runtime contract, external mutable runtime state, and instance-scoped registries.
- Improved the operator surface with stronger managed-wallet reads, token launch-time and price-performance analysis, holder concentration tooling, tracker/news registry support, and better chat/runtime fast paths.
- Cleaned up the public docs and architecture story so the website now explains install, keys, settings, and runtime authority in a much simpler product voice.
- Hardened release packaging with stricter bundle filtering, packaged artifact verification, host-path leak checks, checksum output, smoke testing, compile-workspace pruning, and seed-vault template validation.

## Why This Beta Matters

The last stable release kept pushing TrenchClaw toward better runtime visibility and better release discipline.

`0.1.0-beta.0` takes the next step:

it stops treating those concerns like supporting cleanup and starts presenting them as the product itself.

That means the beta story is now much clearer:

- the runtime owns execution
- the instance owns local state
- tools are the real model-facing contract
- release artifacts are curated, not repo dumps
- docs describe the shipped runtime instead of the repo's history

That is a stronger public surface than simply saying there are more actions than before.

## The Main Themes Of This Release

### 1. The runtime/tool model is much cleaner

One of the biggest changes in this beta is architectural clarity.

The runtime now reads more clearly as:

- a static tool catalog
- a runtime tool snapshot
- lane-aware tool visibility and routing
- chat services that only bind the tools selected for the current turn

That sounds simple, but it is a major product improvement.

When the model-facing contract is clean, three things get better at once:

- prompt quality
- debugging
- operator trust

This beta keeps pushing TrenchClaw away from vague "agent shell" behavior and toward a more explicit local operator runtime.

### 2. Instance-scoped state and settings boundaries are more real

Another major theme in `0.1.0-beta.0` is better separation between shipped runtime contract and live mutable state.

This beta continues the move toward:

- one clear seeded runtime contract under `.runtime`
- one clear mutable runtime root outside normal repo state
- instance-scoped settings, vaults, logs, wallets, workspace files, and registries
- protected settings authority for sensitive runtime paths

That matters because local state discipline is not an implementation detail in this product.

It is one of the main guard rails.

The clearer this boundary gets, the easier it becomes to trust what is local, what is shipped, and what the agent is actually allowed to touch.

### 3. The shipped operator surface is broader and more useful

This release also improves the net shipped operator surface instead of just moving files around.

The current beta now presents a stronger runtime for:

- managed wallet reads and wallet inventory
- token launch-time lookup
- token price-performance analysis
- token holder concentration analysis
- instance-scoped tracker and news-feed registries
- faster runtime/chat paths around common reads

The important point is not "more tools" in the abstract.

The point is that the shipped surface is increasingly oriented around realistic operator workflows: inspect wallets, inspect markets, inspect local runtime state, and move through those flows inside one local runtime.

### 4. The docs and website now tell the right story

This beta also cleans up the public-facing explanation of the product.

The docs now present a simpler model:

- install TrenchClaw
- start the runtime
- sign into an instance
- save the AI key that matters
- understand keys, settings, and architecture only when needed

That is a much better onboarding story than burying users in internals on the first page.

The architecture docs are also cleaner now. They describe runtime authority, chat lanes, tool snapshots, state boundaries, and instance layout in terms that match the current codebase.

That alignment is high leverage.

It reduces support burden, reduces operator confusion, and makes the project feel more intentional.

### 5. Release engineering is now part of the product quality story

One of the most important pieces of this beta is the release surface itself.

This release hardens packaging in several ways:

- bundle filtering stays strict about runtime state, key material, and non-shipping knowledge assets
- release verification checks for blocked files and local machine paths
- packaged artifacts are smoke-tested on the host platform
- checksums and release metadata are written automatically
- compile workspaces are pruned so stray seeded instance data does not ride along into binary builds
- seed vault templates are now validated so shipped runtime seeds cannot silently contain non-empty credentials

That is exactly the kind of work a beta should include.

Before a broader audience uses a local runtime, the release artifacts themselves need to behave like real product artifacts.

## What Operators Actually Get In `0.1.0-beta.0`

After this release, the TrenchClaw story is better described as:

- a local Solana operator runtime with a cleaner, more explicit tool surface
- a runtime that treats instance-scoped state as a real product boundary
- a GUI that sits on top of the runtime instead of competing with it
- a stronger market and wallet inspection surface
- a much clearer install, settings, and architecture story
- a release pipeline that is actively trying not to leak local machine state into shipped artifacts

That is a much stronger beta story than the old repo-shaped explanation.

## What This Release Does Not Pretend

`0.1.0-beta.0` does not pretend TrenchClaw is:

- a fully polished mainstream trading terminal
- a risk-free autonomous trading system
- a finished product surface for every Solana workflow
- a substitute for careful operator review

It does make the current product more coherent.

That is what this beta is for.
